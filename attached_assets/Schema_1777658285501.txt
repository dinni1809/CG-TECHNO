## 1. Overview

CG Techno Electronics v1 operates without a persistent database. All data flows through
serverless API routes and is dispatched via email. Despite this, strict TypeScript schemas
are defined using **Zod** for runtime validation and **TypeScript interfaces** for compile-
time type safety.

This document covers:
- API Payload Schemas (Contact & Application)
- UI Data Schemas (Services, Products, Clients, Testimonials)
- Configuration Schemas
- Email Template Schemas

## 2. API Payload Schemas

### 2.1 Contact / Enquiry Schema

**Purpose:** Validates incoming POST /api/contact request body.

```typescript
// libs/features/src/schemas/contact.schema.ts

import { z } from 'zod';

export const ContactSchema = z.object({
  name: z
    .string()
    .min(2, 'Name must be at least 2 characters')
    .max(100, 'Name must not exceed 100 characters')
    .trim(),

  email: z
    .string()
    .email('Please enter a valid email address')
    .toLowerCase()
    .trim(),

  phone: z
    .string()
    .regex(/^[6-9]\d{9}$/, 'Please enter a valid 10-digit Indian mobile number')
    .trim(),

  message: z
    .string()
    .min(10, 'Message must be at least 10 characters')
    .max(1000, 'Message must not exceed 1000 characters')
    .trim(),

  service: z
    .string()
    .optional(),     // Pre-filled when navigating from service card CTA

  product: z
    .string()
    .optional(),     // Pre-filled when navigating from product enquiry
});

export type ContactPayload = z.infer<typeof ContactSchema>;
```

**Field Reference Table:**

| Field    | Type   | Required | Min | Max   | Validation                             |
|----------|--------|----------|-----|-------|----------------------------------------|
| name     | string | Yes      | 2   | 100   | Alphanumeric + spaces                  |
| email    | string | Yes      | —   | 254   | RFC 5322 email format                  |
| phone    | string | Yes      | 10  | 10    | Indian mobile: starts with 6–9         |
| message  | string | Yes      | 10  | 1000  | No script injection (sanitized)        |
| service  | string | No       | —   | 100   | Passed via query param                 |
| product  | string | No       | —   | 100   | Passed via query param                 |

### 2.2 Application / Careers Schema

**Purpose:** Validates incoming POST /api/apply request body.

```typescript
// libs/features/src/schemas/application.schema.ts

import { z } from 'zod';

export const AVAILABLE_ROLES = [
  'IT Support Engineer',
  'Network Administrator',
  'Sales Executive',
  'Business Development Manager',
  'Intern - IT',
  'Intern - Sales & Marketing',
  'Other',
] as const;

export const ApplicationSchema = z.object({
  name: z
    .string()
    .min(2, 'Name must be at least 2 characters')
    .max(100, 'Name must not exceed 100 characters')
    .trim(),

  email: z
    .string()
    .email('Please enter a valid email address')
    .toLowerCase()
    .trim(),

  phone: z
    .string()
    .regex(/^[6-9]\d{9}$/, 'Please enter a valid 10-digit Indian mobile number')
    .trim(),

  role: z.enum(AVAILABLE_ROLES, {
    errorMap: () => ({ message: 'Please select a valid role' }),
  }),

  message: z
    .string()
    .max(1000, 'Message must not exceed 1000 characters')
    .trim()
    .optional(),

  createdAt: z
    .date()
    .default(() => new Date()),
});

export type ApplicationPayload = z.infer<typeof ApplicationSchema>;
```

**Field Reference Table:**

| Field     | Type   | Required | Min | Max  | Validation                          |
|-----------|--------|----------|-----|------|-------------------------------------|
| name      | string | Yes      | 2   | 100  | Alphanumeric + spaces               |
| email     | string | Yes      | —   | 254  | RFC 5322 email format               |
| phone     | string | Yes      | 10  | 10   | Indian mobile: starts with 6–9      |
| role      | enum   | Yes      | —   | —    | Must be one of AVAILABLE_ROLES      |
| message   | string | No       | —   | 1000 | Cover note / additional context     |
| createdAt | Date   | Auto     | —   | —    | Server-assigned timestamp           |

### 2.3 API Response Schema

**Purpose:** Standardized response shape for all API routes.

```typescript
// libs/features/src/schemas/api-response.schema.ts

export interface ApiSuccessResponse {
  success: true;
  message: string;
}

export interface ApiErrorResponse {
  success: false;
  error: string;
  details?: {
    field: string;
    message: string;
  }[];
}

export type ApiResponse = ApiSuccessResponse | ApiErrorResponse;
```

**Success Example:**
```json
{
  "success": true,
  "message": "Submitted successfully"
}
```

**Validation Error Example:**
```json
{
  "success": false,
  "error": "Validation failed",
  "details": [
    { "field": "email", "message": "Please enter a valid email address" },
    { "field": "phone", "message": "Please enter a valid 10-digit Indian mobile number" }
  ]
}
```

**Server Error Example:**
```json
{
  "success": false,
  "error": "Email service unavailable. Please try again later."
}
```

## 6. Schema Validation Strategy

```
Client Side:
  └── React Hook Form resolver uses Zod schema
  └── Errors shown inline per field on blur/submit

Server Side:
  └── Zod .safeParse() on raw request.json()
  └── On failure: structured 400 response
  └── On pass: sanitize and forward to Nodemailer

Sanitization:
  └── All string fields: .trim() applied in Zod schema
  └── HTML stripped via sanitize-html before email dispatch
  └── No SQL injection risk (no database used)
```
